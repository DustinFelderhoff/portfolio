Story Sundays

Agentic AI Shipping
Local-first AI family storytelling platform · React 19 + Vite 8 + Cloudflare Workers + Supabase
Challenge

My son was born. Grandparents in Texas and Florida had no way to share stories and videos that felt personal, safe, and permanent. Social media meant data leakage, algorithm feeds, and zero ownership. The gap: a private space where AI enhances family storytelling without exposing it outside a closed circle.

Process

Built a full-stack platform from scratch — React 19 frontend, Cloudflare Workers API layer, Supabase Postgres with row-level security, R2 for media storage. The architecture uses a gated TDD pipeline (unit → hooks → integration → e2e:prod) with circuit breakers: if the same error repeats twice, the system stops, researches root cause, and escalates. Three test layers catch what mocks miss. AI transcribes and organizes memories but never exposes them outside the family's contacts window.

Outcome
100%
Gated TDD pipeline adoption
3
Test layers (unit → hooks → int → e2e)
Zero
Data leakage — fully private
  • React 19 + Vite 8
  • TypeScript
  • Cloudflare Workers
  • Supabase / Postgres + RLS
  • R2 Storage
  • Playwright E2E
  • TDD Pipeline
Lesson Learned

A green test suite is not proof a bug is fixed. We shipped bugs that slipped past a fully green mocked test suite — 6 entries in the bug ledger prove it. That's why the project now enforces multi-layer testing with hard gates. A test that passes against mocks but fails in production is worse than no test at all: it gives false confidence.

Lesson Learned

Circuit breakers, not caffeinated retries. When a step fails: root cause first, research the exact error with real version numbers, evaluate each candidate for credibility and blast radius, then apply. Never retry the same error twice without research. This killed the thrash loop problem — the wasted "try X, fail, try Y, fail, try Z" pattern that costs hours.

Autonomous Sub-Agent Orchestration System

Agentic AI System Design
Production-grade multi-agent orchestration with 4 specialized personas, routing logic, and fail-fast boundaries
Challenge

Agent systems that let sub-agents do everything eventually break in unpredictable ways. A researcher with filesystem access overwrites config files. An implementer with web search goes down rabbit holes. The gap: a routing system where agents have strict boundaries, handoff contracts are enforced, and the parent verifies every artifact the child claims to have produced.

Process

Designed four specialized sub-agent personas — researcher, implementer, architect, operator — each with its own toolset whitelist, max iterations, blocked actions, and trigger keywords. Built a routing decision tree that determines whether to handle a task natively, delegate to a single specialist, or orchestrate a mixed workflow. Every delegation boundary follows a handoff contract: the parent sends a self-contained goal + context + toolset, the child returns summary + errors + artifacts + verification, and the parent independently verifies every side effect.

Outcome
4
Specialized agent personas
65
Tools audited across 26 toolsets
Zero
Unverified artifact incidents since shipping
  • Hermes Agent SDK
  • Delegation API
  • Sub-agent Personas
  • Routing Decision Tree
  • Handoff Contracts
Lesson Learned

Never trust a sub-agent's self-report on side-effectful operations. The handoff contract says "the parent verifies." When a child claims it wrote a file or ran a command, the parent reads the file back or stats it. This caught multiple instances where a child "succeeded" but the artifact was empty or misnamed. Verification is not optional — it's the architecture.

Lesson Learned

Tool boundaries are the architecture. The most important design decision wasn't what each agent could do — it was what each agent couldn't do. Blocking the wrong tools prevents entire failure classes. A researcher with filesystem access will eventually overwrite something. An implementer with web search will go down rabbit holes. The boundaries define the system.

Invoice Parser — Intelligent Audit

Agentic AI Zero-to-One Failure Story
First ChatGPT enterprise workflow in 2022 · Python + Prompt Engineering
Challenge

In 2022, our audit operation sent PDF invoices to India to be typed into Excel spreadsheets by hand. Error-prone. Auditors spent 30 hours a week reconciling mistakes. Customer feedback loop was measured in days. ChatGPT had just launched — unproven in enterprise, especially on structured financial data.

Process

Built a ChatGPT-based parser that knew the expected values for each invoice field and extracted them into audit-ready data automatically. Designed prompt/skill system so the model understood expected value ranges and could flag anomalies. One of the earliest enterprise LLM deployments — no playbook existed.

Then I shipped it without training the ops team on the exception workflow. Within two weeks they were flagging every edge case manually. No trust in the AI output. The system was technically correct but operationally broken.

Outcome
30 hrs/wk
Audit time recovered
Days → Hours
Customer feedback loop
Zero
Manual data-entry errors (post-fix)
  • ChatGPT / GPT-3.5
  • Python
  • Prompt Engineering
  • PDF Data Extraction
  • Audit Automation
Hard Lesson — The Fix

Building the right tool isn't enough. The people using it have to trust it. I had to pull the automation back, run three hands-on training sessions with the India team, and redesign the exception workflow with their input. The second launch stuck. Trust took longer to build than the software.

Lesson Learned

Leading the tip of AI means being wrong first. No one had done this before. The skepticism was warranted. What made it work wasn't the technology — it was iterating on the human workflow around it until trust caught up with capability.

System Architecture Optimization & Context Lifecycle

Agentic AI Infrastructure
Four-pillar optimization of a production agent runtime · 65 tools, 26 toolsets
Challenge

A production agent system with 65 tools across 26 toolsets. Context was ballooning, memory was hitting caps, and the system had no automated way to decide what to keep, what to learn, and what to throw away. Without a protocol, every session started fresh — repeating the same mistakes.

Process

Ran a comprehensive architecture audit across four pillars: (1) tool schema refactoring — trimmed verbose descriptions to save token budget, proposed 4 new tool types; (2) context compression & memory decay — built a retention-class system (CRITICAL / DURABLE / SESSION / EPHEMERAL) that auto-decides what to keep and compress at 85% capacity; (3) pre-execution guardrails — 4 silent checks run before every tool call (shell safety, file safety, delegation integrity, sequence integrity); (4) sub-agent orchestration — the routing system described above.

Outcome
4
Optimization pillars delivered
85%
Memory cap auto-compression threshold
4
Pre-execution guardrail checks
  • Tool Schema Design
  • Context Compression
  • Memory Decay Protocol
  • Pre-Execution Guardrails
  • Retention Classification
Lesson Learned

Context management is the unsexy bottleneck of every agentic system. The difference between a tool that works reliably and one that drifts into loops is how well it manages its own context. The retention-class system taught me that most "memory" problems are classification problems — if you know what to keep, what to learn as a skill, and what to throw away, context collapse becomes rare instead of inevitable.

Autonomous Job Application Pipeline

Agentic AI Automation
End-to-end agentic pipeline: discovery → scoring → deep research → voice-captured submission
Challenge

Job hunting is a data problem: hundreds of roles, scattered sources, variable quality, and every application needs to sound like a human. Generic applications don't work. The gap: a system that discovers roles, evaluates fit, researches each company's actual pain points, and generates materials that sound like one specific person.

Process

Built a multi-stage pipeline: Tavily AI Search + SearXNG for discovery, a custom scoring engine with time-decay and location penalties, research sub-agents that investigate each company's competitive gaps, and a voice-of-me gate (canonical transcript → anti-ai-detection scan) that ensures every output sounds human. Single SQLite jobs.db as source of truth. Research workers discover what the company is behind on — not just JD keywords.

Outcome
120+
Job alerts processed per hour
2
Voice gates per application (voice + anti-AI)
Real
Per-company gap research, not JD mirrors
  • Tavily AI Search
  • SearXNG
  • SQLite / jobs.db
  • Research Sub-agents
  • Voice-of-Me Gate
  • Anti-AI Detection
  • Gmail API
Lesson Learned

Voice is the hardest thing to automate, and it's the most important thing to get right. The pipeline can generate a technically perfect letter that still reads like AI. The voice-of-me skill (built from a canonical transcript of how I actually talk) plus anti-ai-detection (catches structural tells like space-dash-space, balanced paragraphs) turned out to be the most valuable components — not the scoring engine. Every piece of copy must pass both gates, in order, or it doesn't ship.

Lesson Learned

A pipeline with no feedback loop is a liability. The location penalty was supposed to downrank non-Remote jobs outside Denver. But if the scraper didn't extract the real location from the JD, the penalty didn't fire. Now every research worker verifies location as a hard gate before scoring. One layer of validation is never enough.

Lessons Learned Across Projects

The patterns that repeat across every system I've built. These are the things I'd tell any PM moving into agentic AI.

1. Trust is the bottleneck, not technology.

Invoice Parser · Story Sundays · Job Pipeline

The Invoice Parser was technically sound the day it shipped. It didn't stick until the ops team trusted it. Story Sundays enforces a gated TDD pipeline because a green mocked suite gave false confidence. In every project, the hardest problem was the same: how do I build something people actually trust, not something that technically works? The answer is always iteration, training, and transparency — never just better code.

2. Boundaries define the system.

Sub-Agent Orchestration · Context Management

The most impactful design decision in the orchestration system wasn't what each agent could do — it was what each agent couldn't do. A researcher with filesystem access will eventually overwrite something. The same rule applies to context management: the retention class system works because it explicitly defines what not to keep. In agentic systems, exclusion is architecture.

3. Never retry the same error twice without research.

Story Sundays · System Architecture

The thrash loop — try X, fail, try Y, fail, try Z — is the most expensive pattern in agentic development. Story Sundays enforces a "same-error-twice" rule: if the normalized error string repeats, stop trying and go to root cause. Research the exact error with real version numbers. Score each candidate solution for credibility (official docs > SO > AI slop). Only then apply. This killed the hours-lost-to-cycling problem cold.

4. AI adoption is a change management problem disguised as a tech problem.

Invoice Parser · Charter (KIRO)

When I brought ChatGPT into an enterprise audit workflow in 2022, the technology was the easy part. The hard part was getting the team to trust the output, building workflows around exceptions, and training people who had been doing it one way for years. Every AI project since has confirmed this: the value of an AI system is multiplied by how well the humans around it can use it, not by how clever the model is.

5. Ship before you're ready, then iterate with discipline.

Invoice Parser · Story Sundays · System Architecture

The Invoice Parser shipped early and broke. Story Sundays ships through a gated pipeline with circuit breakers. The architecture optimization shipped as incremental skills and cron jobs. The pattern: ship fast enough to learn, but gate hard enough to protect the user. The gating mechanisms get more sophisticated with each project, but the first-mover advantage of being early with AI has been consistently valuable.

6. Verification is not optional — not even for sub-agents.

Sub-Agent Orchestration · Job Pipeline

When a sub-agent says "I wrote the file," the parent must stat it and read it back. When a research worker says "I found the data," the pipeline must verify the output format. This sounds obvious but it's the most commonly skipped step — because it's overhead, because trust is faster, because the child usually works. Until it doesn't. Verification at every handoff boundary is now a hard rule in every system I build.
