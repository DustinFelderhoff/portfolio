Lesson Learned

A green test suite is not proof a bug is fixed. We shipped bugs that slipped right past a fully green mocked test suite (6 entries in the bug ledger prove it). That's why the project now enforces a multi-layer test pipeline (unit → hooks → integration → e2e:prod) with hard gates at every stage. A test that passes against mocks but fails in production is worse than no test at all — it gives false confidence.

Lesson Learned

Circuit breakers, not caffeinated retries. When a step fails, the protocol is: root cause first, research the exact error with real version numbers, evaluate each candidate solution for credibility and blast radius, then apply. Never retry the same error twice without research. This killed the thrash loop problem — the wasted pattern of "try X, fail, try Y, fail, try Z" that costs hours.

Autonomous Sub-Agent Orchestration System

Agentic AI System Design
Specialized agent personas with routing logic, handoff contracts, and fail-fast boundaries

Built a production-grade multi-agent orchestration system inside Hermes Agent. Four specialized sub-agent personas — researcher, implementer, architect, operator — each with strict tool boundaries, trigger conditions, and iteration caps. The routing decision tree determines whether to handle a task natively, delegate to a sub-agent, or orchestrate a mixed workflow across multiple specialists.

The key insight: an architect sub-agent can delegate to implementers, but a researcher can't touch the filesystem and an implementer can't browse the web. Every boundary is enforced by the delegate_task schema, not by trust. The handoff contract is explicit about what each child sends back (summary, errors, artifacts, verification) and what the parent must verify independently.

  • Hermes Agent SDK
  • Delegation API
  • Sub-agent Personas
  • Routing Decision Tree
  • Handoff Contracts
Lesson Learned

Never trust a sub-agent's self-report on side-effectful operations. The handoff contract says "the parent verifies." When a child claims it wrote a file, ran a command, or made a network call, the parent reads the file back or stats it. This caught multiple instances where a child "succeeded" but the artifact was empty or misnamed. Verification is not optional.

Lesson Learned

Tool boundaries are the architecture. The most important design decision wasn't what each agent could do — it was what each agent couldn't do. Blocking the wrong tools prevents entire classes of failure. A researcher with access to `write_file` will eventually overwrite something. An implementer with `web_search` will go down rabbit holes. The boundaries define the system.

Invoice Parser — Intelligent Audit

Agentic AI Zero-to-One Failure
ChatGPT-based PDF invoice parser · Python + Prompt Engineering · 2022

In 2022, our audit operation still sent PDF invoices to India to be typed into Excel spreadsheets by hand. The error rate was high, auditors spent 30 hours a week reconciling mistakes, and the customer feedback loop was measured in days. ChatGPT had just launched — unproven in enterprise, especially on structured financial data.

I built a ChatGPT-based parser that knew the expected values for each invoice field and extracted them into audit-ready data automatically. It was one of the earliest uses of LLMs in our enterprise workflow.

Hard Lesson Learned

Building the right tool isn't enough. The people using it have to trust it.

I pushed the parser to production without training the India-based ops team on how to handle exceptions. Within two weeks, they were flagging every edge case manually — no trust in the AI output. The system was technically correct but operationally broken. I had to pull it back, run three hands-on training sessions, and redesign the exception workflow with their input. The second launch stuck.

Lesson Learned

Leading the tip of AI means being wrong first. No one had done this before. There was no playbook for bringing ChatGPT into an enterprise audit pipeline. The skepticism was warranted. What made it work wasn't the technology — it was iterating on the human workflow around it until trust caught up with capability.

System Architecture Optimization & Context Lifecycle

Agentic AI Infrastructure
Four-pillar optimization of a production agent system

Ran a comprehensive architecture audit of the Hermes Agent runtime — 65 tools across 26 toolsets — and delivered an actionable optimization blueprint across four pillars: tool schema refactoring, autonomous context compression and memory decay, pre-execution guardrails, and sub-agent orchestration.

The context compression protocol alone was critical: a retention-class system (CRITICAL / DURABLE / SESSION / EPHEMERAL) that automatically decides what to keep, what to skill-ify, and what to GC. When the memory store hits 85% capacity, it consolidates and compresses instead of failing. The pre-execution guardrails (shell safety, file write safety, delegation integrity, sequence integrity) run silently before every tool call and prevent the most common failure classes before they happen.

  • Tool Schema Design
  • Context Compression
  • Memory Decay Protocol
  • Pre-Execution Guardrails
  • Cron Health Pipeline
Lesson Learned

Context management is the unsexy bottleneck of every agentic system. The difference between a tool that works reliably and one that drifts into loops is how well it manages its own context. The retention-class system taught me that most "memory" problems are classification problems — if you know what to keep, what to learn as a skill, and what to throw away, context collapse becomes rare instead of inevitable.

Autonomous Job Application Pipeline

Agentic AI Automation
End-to-end agentic pipeline: discovery → scoring → research → voice capture → submission

Built a fully autonomous job application system: Tavily AI Search + SearXNG for discovery, a custom scoring engine (time-decay factored, location penalties for blocker cities), per-job deep research via sub-agents, voice-captured cover letter generation, and ATS-aware resume building. The pipeline processes 120+ job alerts hourly, runs research workers on top candidates, and generates application-ready materials.

The key architecture decisions: a single SQLite `jobs.db` as the canonical source of truth, sub-agents for parallel research, and a skill-based pipeline (voice-of-me → anti-ai-detection gate) that ensures every output sounds like a human, not an LLM. The research worker discovers company-specific pain points and competitive gaps — not just JD keywords — so every cover letter has genuine specificity.

  • Tavily AI Search
  • SearXNG
  • SQLite / jobs.db
  • Sub-agent Research Workers
  • Voice-of-Me Gate
  • Anti-AI Detection
  • Gmail API
Lesson Learned

Voice is the hardest thing to automate, and it's the most important thing to get right. The pipeline can generate a technically perfect cover letter that still reads like AI. The voice-of-me skill (built from a canonical transcript of how I actually talk) plus the anti-ai-detection gate (catches structural tells like space-dash-space, balanced paragraphs, rule-of-three) turned out to be the most valuable components — not the scoring engine or the research workers. Every piece of copy must pass both gates, in order, or it doesn't ship.

Lesson Learned

A pipeline with no feedback loop is a liability. The scoring engine had a location penalty that was supposed to downrank non-Remote jobs outside Denver. But it was only as good as the data it ran on — if the scraper didn't extract the real location from the JD, the penalty didn't fire. Now every research worker verifies location as a hard gate before scoring, and location_penalty() is enforced in two layers. One layer of validation is never enough.

Lessons Learned Across Projects

1. Trust is the bottleneck, not technology.

Applied in: Invoice Parser, Job Pipeline, Story Sundays

The Invoice Parser was technically sound the day it shipped. It didn't stick until the ops team trusted it. Story Sundays enforces a gated TDD pipeline because a green mocked suite gave false confidence. The job pipeline has a two-layer validation gate because one layer was never enough. In every project, the hardest problem was the same: how do I build something people actually trust, not something that technically works? The answer is always iteration, training, and transparency — never just better code.

2. Boundaries define the system.

Applied in: Sub-Agent Orchestration, Context Management

The most impactful design decision in the orchestration system wasn't what each agent could do — it was what each agent couldn't do. A researcher with filesystem access will eventually overwrite something. An implementer with web search will rabbit-hole. The same rule applies to context management: the retention class system (CRITICAL / DURABLE / SESSION / EPHEMERAL) works because it explicitly defines what not to keep. In agentic systems, exclusion is architecture.

3. Never retry the same error twice without research.

Applied in: Story Sundays, System Architecture

The thrash loop — try X, fail, try Y, fail, try Z — is the most expensive pattern in agentic development. Story Sundays enforces a "same-error-twice" rule: if the normalized error string repeats, stop trying and go to root cause. Research the exact error with real version numbers. Score each candidate solution for credibility (official docs > SO > AI slop). Only then apply. This killed the hours-lost-to-cycling problem cold.

4. AI adoption is a change management problem disguised as a tech problem.

Applied in: Invoice Parser, Charter (KIRO)

When I brought ChatGPT into an enterprise audit workflow in 2022, the technology was the easy part. The hard part was getting the team to trust the output, building workflows around exceptions, and training people who had been doing it one way for years. Every AI project since has confirmed this: the value of an AI system is multiplied by how well the humans around it can use it, not by how clever the model is.

5. Ship before you're ready, then iterate with discipline.

Applied in: Invoice Parser, Story Sundays, System Architecture

The Invoice Parser shipped early and broke. Story Sundays ships through a gated pipeline with circuit breakers. The architecture optimization shipped as skills and cron jobs that could be applied incrementally. The pattern: ship fast enough to learn, but gate hard enough to protect the user. The gating mechanisms get more sophisticated with each project, but the first-mover advantage of being early with AI (2022 ChatGPT, early agentic orchestration, autonomous pipelines) has been consistently valuable.

6. Verification is not optional — not even for sub-agents.

Applied in: Sub-Agent Orchestration, Job Pipeline

When a sub-agent says "I wrote the file," the parent must stat it and read it back. When a research worker says "I found the data," the pipeline must verify the output format. This sounds obvious but it's the most commonly skipped step — because it's overhead, because trust is faster, because the child usually works. Until it doesn't. The handoff contract now explicitly requires artifact verification, and skipping it is a lint-level violation.
