Lessons Learned Across Projects

Patterns I keep seeing across every system I build. Worth knowing if you're moving into agentic AI.

1. Trust matters more than the tech stack.

Invoice Parser · Story Sundays · Job Pipeline

The Invoice Parser was technically sound the day it shipped. It didn't stick until the ops team trusted it. Story Sundays enforces a gated TDD pipeline because a green mocked suite gave false confidence. In every project, the hardest problem was the same: how do I build something people actually trust, not something that technically works? The answer is always iteration, training, and transparency, never just better code.

2. What you block matters more than what you allow.

Sub-Agent Orchestration · Context Management

The most impactful design decision in the orchestration system wasn't what each agent could do. It was what each agent couldn't do. A researcher with filesystem access will eventually overwrite something. The same rule applies to context management: the retention class system works because it explicitly defines what not to keep. Deciding what to leave out did more for reliability than any feature I added.

3. Never retry the same error twice without research.

Story Sundays · System Architecture

The thrash loop, try X, fail, try Y, fail, try Z, is the most expensive pattern in agentic development. Story Sundays enforces a "same-error-twice" rule: if the normalized error string repeats, stop trying and go to root cause. Research the exact error with real version numbers, then score each candidate solution for credibility (official docs > SO > AI slop) before you apply it. This killed the hours-lost-to-cycling problem.

4. AI adoption is really a change management problem.

Invoice Parser · Charter (KIRO)

When I brought ChatGPT into an enterprise audit workflow in 2022, the technology was the easy part. The hard part was getting the team to trust the output, building workflows around exceptions, and training people who had been doing it one way for years. Every AI project since has confirmed this: the value of an AI system is multiplied by how well the humans around it can use it, not by how clever the model is.

5. The tension between speed and safety never goes away.

Invoice Parser · Story Sundays · System Architecture

The Invoice Parser shipped early without the right gating. It created distrust that took weeks to undo. Story Sundays swung too far the other way: a gated pipeline that slowed learning. The pattern that emerged across projects isn't about picking one side. It's that the gating mechanisms get smarter with each project, not stricter. The Invoice Parser had no gates. Story Sundays has a gated pipeline with circuit breakers. The architecture optimization has skills and cron jobs that ship incrementally. The thing I keep relearning: ship early enough to learn what actually matters, but protect the user hard enough that they keep trusting you after.

6. Verify everything, including your own sub-agents.

Sub-Agent Orchestration · Job Pipeline

When a sub-agent says "I wrote the file," the parent must stat it and read it back. When a research worker says "I found the data," the pipeline must verify the output format. This sounds obvious but it's the most commonly skipped step, because it's overhead, because trust is faster, because the child usually works. Until it doesn't. Verification at every handoff boundary is now a hard rule in every system I build.
